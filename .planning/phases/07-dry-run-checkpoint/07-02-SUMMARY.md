```json
    "ghost_hash": "sha256:<64 hex>",
    "timestamp": "2026-04-04T21:10:33.000Z",
    "ccaudit_version": "0.0.1",
    "checkpoint_version": 1
  }

```

**Key invariants Plan 03 can assert against:**
- `meta.exitCode === 0` always (D-03 â€” dry-run success never sets exit 1)
- `dryRun === true` is the discriminator between dry-run and regular ghost envelopes
- `changePlan.archive`, `changePlan.disable`, `changePlan.flag` are always arrays (may be empty)
- `changePlan.counts` has exactly the 4 keys: `agents, skills, mcp, memory`
- `changePlan.savings.tokens` is a raw integer (not the `~Xk` display form)
- `checkpoint.ghost_hash` has the literal `sha256:` prefix
- `checkpoint.checkpoint_version === 1` (integer, not string)
- `checkpoint.ccaudit_version` is a non-empty string from the generated `_version.ts`
- `meta.timestamp === checkpoint.timestamp` in practice â€” both are `new Date().toISOString()` called in the same tick (Plan 03 should tolerate millisecond drift)

Note: the `changePlan` field mirrors the full ChangePlan shape from `@ccaudit/internal`, and the `checkpoint` field is a compact projection of the persisted checkpoint (the full 7-field D-17 schema lives in the on-disk file; the envelope carries 5 of those 7 fields â€” no `item_count` duplication since it's equivalent to `changePlan.counts`, and no `since_window` duplication since it's already in `meta.since`).

## Decisions Made

1. **Single-decision-point dry-run branch (placement)**: The plan's RESEARCH Â§CLI Integration section mandates "lift the decision earlier, one test path per output mode per command mode". The branch is placed immediately after `enrichScanResults` (which produces the `TokenCostResult[]` that `buildChangePlan` consumes) and before `calculateHealthScore` (inventory-only). This keeps the scan/enrich pipeline single-source and guarantees that any future changes to discovery/parsing/scanning automatically flow through both branches.

2. **CCAUDIT_VERSION via prebuild/pretest hook vs tsdown `define`**: tsdown's `define` option is already used for `import.meta.vitest â†’ undefined` (stripping test code). Adding a second define for the version would tie the version value to build-config rather than to package.json. The script+hook pattern means `pnpm --filter ccaudit test` AND `pnpm --filter ccaudit build` both regenerate the constant before running â€” so local dev, CI, and release all read from the single source of truth (`package.json`) at the right time. The generated file is git-ignored to prevent accidental hand-edits.

3. **Stale dist .d.ts rebuilds during typecheck (documented as Rule 3 blocking fix)**: `packages/internal/dist/index.d.ts` and `packages/terminal/dist/index.d.ts` were stale from before Plan 01. Project references in TypeScript consume these declaration files, so the new `ChangePlan` type (Plan 01) and `renderChangePlan` export (Plan 02 Task 1) were invisible to downstream `tsc --noEmit` until I ran `pnpm --filter @ccaudit/internal exec tsc` and `pnpm --filter @ccaudit/terminal exec tsc`. These rebuilds are NOT committed (dist/ is gitignored) but they must be run once to unstick the typecheck chain. Recommend adding a `pnpm -r exec tsc` to Plan 03's verification or post-commit hook.
