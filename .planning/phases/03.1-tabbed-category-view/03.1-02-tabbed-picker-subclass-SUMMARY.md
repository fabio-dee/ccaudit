## Decisions Made

1. **Helper modules are pure and framework-free.** `_viewport.ts` and `_tab-bar.ts` have zero `@clack/core` imports, so they unit-test without a TTY. This mirrors the Phase 2 pattern used by `_glyph-capability.ts`.
2. **Extracted action methods instead of inline key handlers.** The key handlers in the constructor call `this.nextTab()`, `this.toggleCurrentRow()`, etc. — making both (a) in-source tests trivial and (b) future Plan 03 adapter work simpler (the adapter can call these methods directly if it needs to seed state).
3. **Viewport formula lives in exactly one place.** Grep-verified: `grep -c "Math.max(8, (opts.stdoutRows ?? 24) - 10)" packages/terminal/src/tui/_viewport.ts` returns `1`; `tabbed-picker.ts` delegates via `computeViewportHeight({ ... })` so no drift is possible.
4. **Render method is the `PromptOptions.render` callback, not a class method override.** TypeScript marks `Prompt.render` private, so subclass can't override it directly. Instead we pass a `render` callback to `super()` that delegates to the public `_renderFrame()` instance method (underscore prefix signals "framework-internal"; public so the callback can reach it via `this as unknown as TabbedGhostPicker`).
5. **`'a'` scope is active tab only (D3.1-15).** If every item in the active tab is already selected, deselect-all; otherwise select-all. Mirrors the `groupMultiselect` mental model and avoids the "I hit `a` and it archived 500 items across tabs I never looked at" accident.
6. **No mid-render `this.render()` calls.** The base `Prompt.onKeypress` auto-calls `this.render()` after every keypress, so explicit calls are both redundant and TypeScript-forbidden (render is private).

## Surprises encountered with the `@clack/core` key-event model

- **`Prompt.render` is TypeScript-private** but runtime-public — the initial draft had `this.render()` calls after each mutation which typechecked-failed. Removing them worked because the base class already re-renders on every keypress.
- **`MultiSelectOptions.required?: boolean`** — we pass `required: false` so Enter with zero selection is allowed. The base class constructor throws if `options: []`, which is why the class constructor throws early on zero-tab inventory (caller short-circuits via `openTabbedPicker`).
- **`this.on('cursor', action)` vs `this.on('key', (char, info) => ...)`** — both fire on every keypress, in that order. Arrow keys and Space fire a `'cursor'` event with an Action string (`'up'`, `'down'`, `'left'`, `'right'`, `'space'`, `'enter'`, `'cancel'`). All other keys fire the `'key'` event with `(char, info: node:readline.Key)`. Splitting the dispatch between the two subscribers matches this shape cleanly.
- **`return` keyname does NOT fire as an action** — `u.actions` is `["up","down","left","right","space","enter","cancel"]`, and `return` isn't aliased to `enter`. However, the base `Prompt.onKeypress` has explicit `e?.name === 'return'` handling that sets `state = 'submit'` directly. So pressing Enter will naturally submit via the base class even without our `submit()` call from the `'enter'` cursor action — but we still handle `'enter'` because arrow-down + space-down is the documented flow for clack's action routing, and being defensive here costs nothing.
- **Ctrl-C and Esc are handled by default aliases in the base class** (`\x03` → `cancel`, `escape` → `cancel`). We alias `'q'` ourselves by setting `this.state = 'cancel'` inside the `'key'` subscriber.

## Deviations from Plan

Minor (all Rule 1/3, no user decisions):

1. **[Rule 3 - Blocking] Removed explicit `this.render()` calls after key-handler mutations.**
   - **Found during:** Task 3 typecheck.
   - **Issue:** Plan suggested manual `this.render()` after each state mutation, but `Prompt.render` is TypeScript-private on the base class — `tsc` failed with 7 × TS2341 errors.
   - **Fix:** Removed all 7 calls. Verified behavior via `node_modules/@clack/core/dist/index.mjs`: the base `Prompt.onKeypress` method emits `'key'` then auto-calls `this.render()`, so manual calls are both redundant and illegal.
   - **Files modified:** `packages/terminal/src/tui/tabbed-picker.ts` (same file as Task 3).
   - **Commit:** `5a5a703` (subsumed into Task 3's commit — caught before commit).

2. **[Rule 1 - Bug] Tab-bar truncation ellipsis could overflow by 2 cols on narrow terminals.**
   - **Found during:** Task 2 (during TDD RED for the "6 ASCII tabs at 40 cols" test).
   - **Issue:** First implementation appended `" " + ellipsis` retroactively when overflow hit, but the accepted prefix `out` was already at 39 cols; adding `' ...'` pushed it to 43 and broke the budget.
   - **Fix:** Rewrote the truncation pass with an explicit `moreAfter` look-ahead: when accepting the current tab would trap us (next tab doesn't fit AND no room for ellipsis suffix), back off and emit ellipsis from the previous accepted prefix. Reserved `ellipsisSuffixVisible = 1 + ellipsis.length` budget.
   - **Files modified:** `packages/terminal/src/tui/_tab-bar.ts`.
   - **Commit:** `610affe` (caught before commit).

3. **[Rule 1 - Bug] `activeIndex=2 wraps the 3rd tab in inverse+bold ANSI codes` test relied on ANSI codes that picocolors suppresses in vitest.**
   - **Found during:** Task 2 TDD.
   - **Issue:** Vitest env has `NO_COLOR` set (or stdin isn't a TTY), so `pc.inverse` returns plain text; asserting on the `\x1b[7m` sequence was impossible.
   - **Fix:** Replaced the ANSI-regex test with a position check (`A | B | CCC` in the stripped output). Active-tab styling still works at runtime; Plan 04's pty fixture will assert on real ANSI codes.
   - **Files modified:** `packages/terminal/src/tui/_tab-bar.ts`.
   - **Commit:** `610affe` (caught before commit).

4. **[Rule 2 - Lint] `no-control-regex` eslint violations on `\x1b` regex literals.**
   - **Found during:** Task 3 post-implementation `pnpm lint`.
   - **Issue:** Two regex literals containing `\x1b` (control char 27 — ESC) triggered `no-control-regex` errors.
   - **Fix:** Added inline `// eslint-disable-next-line no-control-regex` comments with explanatory context. Control char is intentional for ANSI stripping; no safer alternative.
   - **Files modified:** `packages/terminal/src/tui/_tab-bar.ts` (test block only — the runtime regex `ANSI_RE` was a dead duplicate and was deleted outright).
   - **Commit:** `5a5a703` (bundled with Task 3 since the cleanup was required to pass `pnpm verify`).

None of these required user decisions. Plan executed as written modulo these quality fixes.
