## Decisions Made

1. **Removed `_clack` injection entirely (not renamed).** Grep confirmed no production caller uses `_clack` — only the in-source tests in `select-ghosts.ts` did, and this plan rewrote those tests. Swapping to `_picker: PickerDep` gives tests a direct seam at the new call path.
2. **D3.1-16 gate uses `process.exit(1)` not a returned outcome.** The context note says exit 1 explicitly. Using a returned sentinel would broaden `SelectGhostsOutcome` beyond the Phase 2 contract and break SC7. Plan 04 integration test will assert the gate by spawning a subprocess with `COLUMNS`/`LINES` overrides.
3. **CATEGORY_ORDER / CATEGORY_LABEL / formatRowLabel home = `select-ghosts.ts`** (not a new `_category-labels.ts`). Plan 02's summary explicitly recommended this: "Plan 03 should export them from `select-ghosts.ts` (they're already exported there for `formatRowLabel`) and remove the duplicated constants from `tabbed-picker.ts`." Following that recommendation verbatim.
4. **Did NOT modify `ghost.ts`.** Task 3 was a verification-only task. Hash `ba641c942a08bdfeafa386a6809858ce547a5a38` before and after. `selectGhosts(` still returns 1 grep hit, matching Phase 2's call-site count.

## Observations Affecting Plan 04's pty-fixture design

- **`ghost.ts` call path unchanged.** The pty fixtures can use the existing `_test-helpers.ts` pty helpers that Phase 3 established; no new entry points needed. Test invocations via `runCcauditGhost` spawning the CLI with `CCAUDIT_FORCE_TTY=1` + a pty still produce the same observable behavior — they now land in `openTabbedPicker` instead of `groupMultiselect`.
- **Terminal-too-short gate is subprocess-observable.** Plan 04's fixture can assert `exit code 1` + `stderr matches /Terminal too short \(need ≥14 rows, got 10\)/` by launching the CLI with `LINES=10` in the env. The child process will write exactly the message (verified by grep in `select-ghosts.ts`) and exit 1 before any checkpoint/manifest is written. This also re-confirms INV-S2 invariance under the new picker (SIGINT before gate is tested separately; the gate itself is a different code path).
- **`openTabbedPicker` is the single point where the base `@clack/core.MultiSelectPrompt.prompt()` is invoked now.** Plan 04's SIGINT-cancel test should drive the pty against the merged path: `selectGhosts({...})` → `openTabbedPicker({...})` → `new TabbedGhostPicker(...).prompt()` → SIGINT → `{ kind: 'cancel' }` → adapter passes through → `ghost.ts` stderr "No changes made." + exit 0. If INV-S2 held in Phase 2's `groupMultiselect` flow it holds here since the cancel short-circuit is identical shape.
- **Cross-tab selection state lives on the `TabbedGhostPicker` instance** (as documented in Plan 02). Plan 04's test for "select 2 AGENTS, switch to SKILLS, select 1, switch back, press Enter → 3 IDs in manifest" should pass without additional plumbing — the adapter is a pure pass-through of `picker.selectedIds`.
- **PickerDep is available for adapter-level tests in apps/ccaudit.** If Plan 04 wants to smoke-test `runInteractiveGhostFlow` with a mock picker (e.g. to avoid pty spawn overhead for some invariant tests), it can `import { PickerDep } from '@ccaudit/terminal'` and pass `_picker: fake` — the new seam is intentionally public via the barrel.
