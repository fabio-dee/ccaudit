# Phase 03.2 Plan 01: Change-plan commands branch Summary

**One-liner:** Close Bug #1 at the domain layer — wire the `command` category end-to-end through buildChangePlan, ArchiveOp, archiveOne, restore, and both confirmation renderers (TUI + non-interactive text), making counts.commands a required ChangePlan field.

## Objective

Close Bug #1 (commands silently dropped by `buildChangePlan`) by:

1. Adding `commands: number` as a REQUIRED field on `ChangePlan.counts`
2. Adding a `category === 'command'` branch to `buildChangePlan` that archives definite-ghost commands
3. Mirroring the same addition in `filterChangePlan`
4. Widening `ArchiveOp.category` and `buildArchiveOp`'s input to accept `'command'`
5. Teaching `archiveOne` + `findCategoryRoot` to resolve the `commands` path segment
6. Extending `BustCounts.archive` with a `commands` counter wired through `runBust` and the manifest footer
7. Adding a command-unarchive step to `restore.ts`
8. Rendering a commands row in the interactive confirmation screen (both Unicode and ASCII branches)
9. Rendering a commands row in the non-interactive `renderChangePlan` ARCHIVE group (unblocks Plan 05 SC3)
