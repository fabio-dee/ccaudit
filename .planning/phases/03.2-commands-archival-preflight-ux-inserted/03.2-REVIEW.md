## Warnings

### WR-01: `runPreflightRetryLoop` retry iteration pid drift not covered by tests

**File:** `packages/terminal/src/tui/_preflight-copy.ts:117-144`

When `initialResult` is provided with `selfInvocation=false`, the loop writes its copy, then on user-confirmed retry re-detects via `detectFn`. Between prompt-confirm and re-detect, the caller's original `initialResult.pids` may be stale vs. the new detection. Behavior is correct — the fresh `detectFn` supersedes the initial — but the invariant isn't covered by tests. Not a bug, but a documentation/test-coverage weakness.

**Fix:** Add a test case that asserts on retry iteration 2 the rendered pids come from `detectFn` output, not from the stale `initialResult`.

### WR-02: Entry preflight detect counter semantics are subtle

**File:** `apps/ccaudit/src/cli/commands/ghost.ts:252-291`

The entry preflight at line 253 calls `detectClaudeProcesses` directly, constructs `initialResult`, then passes it to `runPreflightRetryLoop`. Inside the loop, on user-confirmed retry, `detectFn` is invoked again — so the initial detection path consumes one DIRTY-counter tick at line 253, then one more on each retry. A reader expecting `CCAUDIT_TEST_PREFLIGHT_DIRTY=2` for "2 retries" will be off-by-one. The per-layer counter semantics are correct but subtle.

**Fix:** Add an inline comment at line 253 explicitly noting "this initial detect consumes one DIRTY-counter tick; each confirmed retry consumes one more."

**File:** `packages/terminal/src/tui/confirmation.ts:96-97, 136`

**Fix:** Add parallel singular/plural labels for agents/skills/commands/memory consistent with the existing `mcpLabel` pattern, and update the locked tests.
